AI Agents — Architecture, Tools & Operations

Reference for builders: definitions, system design, workflow automation, API patterns, collaboration models, and production habits. Keep language models grounded with tools, memory, and clear objectives.

1. What Are AI Agents

Definition. An agent is a system that perceives context (user input, state, tool results), reasons toward a goal, and acts—often by calling tools, updating memory, or delegating work—until stopping criteria are met. It is not just a one-shot completion; it is a loop (plan → act → observe).

Contrast with a static chat: agents bind the model to environment, policies, and feedback.

Type Role
Reactive Maps input → action with little long-term state; good for classifiers, routers, and guardrails.
Deliberative Builds a plan, tracks subgoals, revises on failure; common in research and coding copilots.
Tool-using Extends the model with APIs, SQL, search, and browsers—most production "agents" are this class.
Multi-agent Multiple roles (planner, coder, critic) with orchestration, shared memory, or handoffs.

Capabilities to design for: retrieval, structured output, idempotent tools, human approval on sensitive steps, and measurable success criteria (accuracy, cost, latency).

Situation Prefer
One fixed transform (e.g. summarize ticket) Single LLM call or small chain—skip agent state.
Search + act across systems Tool-using agent with retrieval and an explicit budget (rounds, cost).
Open-ended research or coding Multi-step agent with checkpointing, tests, and optional human review.
Regulated or irreversible side effects Human-in-the-loop and dry-run or staging tools first.

Reality check: "Agent" is often marketing. If your flow never branches on tool output, you likely have a pipeline—not an agent. Name systems by behavior, not hype.

2. Agent Architecture

A minimal loop: perceive (messages + RAG) → plan (optional CoT or planner model) → act (tools) → update memory → repeat or terminate.

  • Planning: Decompose the user goal, choose tools, set checkpoints. Use explicit "next step" prompts or a dedicated small model for triage to reduce main-model cost.
  • Memory: Short-term = conversation + scratchpad. Long-term = vector DB, documents, or entity store. Version snapshots so you can roll back bad writes.
  • Tool use: JSON-schema tools, least privilege, timeouts, and retries with idempotency keys for writes.
  • Reasoning: Chain-of-thought (internal or summarized), self-critique, or a "verifier" pass before irreversible calls.
Component What to get right
Context window Trim history; summarize; inject only top-k retrieved chunks; avoid poisoned or stale RAG.
State machine Encode allowed transitions (e.g. draft → review → send). Block tools outside the current state.
Policies PII redaction, rate limits, tool allow lists, and escalation to humans for high-risk actions.
Control loop (conceptual)
while not done and round < max_rounds:
  ctx = build_context(user_msg, memory, retrieved_docs)  # RAG optional
  plan_or_action = model(ctx, available_tools=TOOLS)
  if plan_or_action.type == "tool":
      result = run_tool_safely(plan_or_action)  # validate, timeout, log
      memory.append(result)
  elif plan_or_action.type == "final":
      return plan_or_action.answer
  round += 1
raise or escalate  # out of budget or stuck

RAG in agents: chunk + embed with provenance, filter by access control, re-rank, then attach citations to the model. Refresh embeddings when source docs change. Measure "groundedness" (answer supported by chunks).

Memory layer Good for
Buffer / window Current task details; compress older turns to summaries.
Vector + metadata Procedures, long docs; pair with source IDs and ACLs.
Key-value (CRM/DB) Authoritative user facts; avoid duplicating in free text.

3. Building with n8n

n8n is a node-based workflow engine. Use it to wire triggers (webhooks, schedules, apps) to AI steps and downstream systems without shipping a full custom backend for every integration.

  1. Setup: Self-host (Docker) or n8n Cloud. Store credentials in n8n’s vault; never hard-code keys in node expressions.
  2. Workflows: One workflow per use case. Use Error Workflow to centralize failure alerts (Slack, email, PagerDuty).
  3. Triggers: Webhook for synchronous APIs; Cron for batch; Queue triggers (e.g. Redis) for backpressure. Add idempotency keys for retried deliveries.
  4. AI nodes: OpenAI / Anthropic / LangChain sub-nodes, or HTTP Request to your gateway. Keep prompts in separate nodes for diffability. Pass structured JSON between nodes.

Pattern: Webhook → validate payload → (optional) fetch context (HTTP/DB) → AI node with tools via sub-workflow or function node → map response → respond / notify. Rate-limit the webhook in your reverse proxy.

Self-host hardening: TLS at edge, BASIC_AUTH or SSO for admin UI, private network for DB, disable execution if you only need static hosting of definitions, and pin image digests. Back up ~/.n8n and credentials with your DR plan.

Node / area Agent-related use
Code / Function Transform LLM output into tool payloads; validate with Zod/Pydantic-style checks.
Execute Workflow Nest reusable "tool" subflows (e.g. create ticket) with one place to add retries.
Wait / If / Switch Branch on model confidence, user approval webhooks, or business rules.
HTTP Request / GraphQL Call your own agent API, embedding service, or vector DB. Set sane timeouts (10–30s typical).
Sticky / Queue Smooth spikes; one long workflow run per user message in async mode.

4. Tool Calling & Function Calling

Models emit tool calls (name + arguments) that your app executes. Reliability comes from strict schemas, validation, and small tool surfaces per step.

Provider pattern Practical tips
OpenAI tools / functions Attach tools with JSON Schema; use parallel_tool_calls when safe; return tool results in the next user/tool message. Prefer strict schema when available.
Anthropic tool use Tools block in API; assistant returns tool_use content; you append tool_result user messages. Good for interleaved text + tool calls.
Cross-vendor Abstract tool registry + adapter layer. Log raw calls for replay. Enforce max rounds and token budgets. Never trust the model to skip validation.

Anti-patterns: giant catch-all tools; allowing raw SQL or shell from the model; returning secrets in tool error strings; unbounded re-invocation on the same error.

ReAct-style flows: interleave thought (optional, may omit in prod), action (tool), observation (result). In APIs this maps to multiple chat turns with tool messages—your runtime stitches the loop. Prefer storing only action + observation in logs to reduce PII in chain-of-thought text.

Schema practice Why
Concrete enums over free strings Reduces bad tool names and eases static analysis in CI.
Required fields + max lengths Prevents runaway args and OOMs in downstream services.
Narrow return payloads Tools return evidence not whole tables; paginate in the app layer.
Idempotency keys (writes) Safe retries if the model or network duplicates a call.

5. Multi-Agent Systems

Split work when roles and expertise differ: orchestrator decomposes; specialists execute; critic / reviewer reduces hallucination before deliverables ship.

  • Orchestration: Central planner (single brain) vs. peer debate—pick central for auditability, peer for creative divergence with a merge step.
  • Sub-agents: Narrow prompts and tools per agent; pass compact handoff packets (goal, constraints, files, done criteria) instead of full chat logs.
  • Delegation: Define escalation: when to ask a human, when to retry, when to fail closed. Use a shared task queue or blackboard for state.

Watch: cost explosion (N×rounds), inconsistent world model across agents, and duplicate tool side effects. Mitigate with shared schemas, one writer per resource, and execution traces.

Pattern When it shines
Supervisor → workers Single routing decision point; good trace for compliance and debugging.
Sequential handoff Pipeline: research → draft → review; each stage has different tools and budgets.
Parallel + merge Compare approaches (A/B) or sources; reducer step must be deterministic.
Critic / judge Second pass on code or policy; keep rubric short to avoid dueling models forever.

6. Popular Frameworks

Framework Strengths & fit
LangChain / LangGraph Chains, RAG, tools; LangGraph adds stateful graphs, cycles, and human-in-the-loop—good for controllable multi-step agents in Python/JS.
CrewAI Role-based crews with processes (sequential, hierarchical). Fast to prototype "team" automations; watch token burn and overlapping responsibilities.
AutoGen / AG2 Conversable agents, group chat patterns, and code execution sandboxes. Strong for research-style pipelines; secure execution environment is a must.
n8n Low-code ops glue: schedules, webhooks, CRM, email. Pair with a small "brain" service when you outgrow pure no-code control flow.
LlamaIndex / Haystack Retrieval-first stacks: indexing, query engines, agents over docstores—great when quality depends on data ingestion more than model choice.
Semantic Kernel .NET-oriented planners and plugins; fits enterprise LOB systems and strong typing for tools.
Mastra, OpenAI SDK Lightweight when you need custom TypeScript/Node agents without the full kitchen sink—own your abstractions and tests.

Choose by language, operational maturity (observability, testing), and whether you need arbitrary code in the hot path. Frameworks are accelerators, not substitutes for data and policy design.

7. Deployment & Monitoring

  • Hosting: Serverless (Vercel, Lambda) for bursty I/O; containers (Fly, K8s) for long-lived workers and websockets. Keep model calls behind a gateway for routing, fallbacks, and key rotation.
  • Logging: Structured JSON logs: request_id, user_id, model, tool_name, latency, tokens, error_code. Store prompts/results with PII redaction; retain per compliance needs.
  • Traces: OpenTelemetry spans for each model + tool call. Visualize slow steps and failed chains—optimize what you measure.
  • Error handling: Classify: retryable (429, 5xx with backoff) vs. fatal (schema mismatch). Return safe user messages; never leak stack traces or secrets. Circuit-break persistently failing tools.
Signal Why it matters
Token & cost / request Detect prompt bloat, runaway tool loops, and model drift after upgrades.
Tool failure rate Flag upstream API issues or bad schemas before users churn.
Human override rate High rate means unclear UX or over-trusted automation—tune policies and prompts.
Qualitative / eval scores LLM-as-judge or rubrics in CI to catch regressions on golden prompts when models update.
Queue depth & p95 latency Surface overload early; add concurrency caps per tenant before providers throttle you.

SLOs: Set separate targets for interactive (e.g. first token 500ms) vs. background jobs. Alert on SLO burn with multi-window detectors to avoid flappy pages.

Kill switches: per-tool disable in config; global "model off" for incidents; feature flags for prompt versions so rollback is a toggle, not a deploy.

Reference: Core Terms

System prompt Stable instructions, persona bounds, and safety rules. Version in git; avoid stuffing secrets (they can leak in errors).
Tool / function Named capability with a schema. Your code executes; the model only proposes invocations—you are the security boundary.
Stop sequences Tokens that halt generation. Rare in modern tool APIs, but still useful in raw text pipelines.
Temperature Lower for structured tools and JSON; higher for brainstorm-only steps. Some stacks fix temperature per step.
Context rot Quality drop as the window fills with tool noise. Mitigate: summarize, drop redundant observations, and reset state.
Eval harness Repeatable test cases: expected tool calls, no forbidden tools, and answer checks. Run nightly or in CI for prompt PRs.
PII & data residency Classify data before it hits the model. Route regulated workloads to region-pinned providers and log minimization.
