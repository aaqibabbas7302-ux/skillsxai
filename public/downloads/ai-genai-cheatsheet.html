AI & Generative AI Cheat Sheet

At a glance

One-page map of core concepts used in product ML, LLM features, and GenAI system design. Pair with your vendor’s model cards, pricing, and ToS. Terms like “GPT / Claude / Gemini / Llama” are families of models that evolve; always check current capabilities.

1. AI Fundamentals

Core branches of modern AI: how data, objectives, and architecture connect to the task (classify, predict, generate, plan).

Machine Learning (ML)

Learn functions from data so the program improves with examples. Types: supervised (labeled pairs), unsupervised (structure/density), semi-supervised, and reinforcement (trial + reward). Distinct from rule engines: generalization, not if/else hand-tuning at scale.

Deep Learning (DL)

ML with stacked parametric layers (neurons with weights). Backpropagation + optimizers (Adam, SGD) minimize a loss. Needs data volume and compute; overfitting is when the model memorizes training noise—combat with regularization, more data, dropout, early stopping, better validation design.

Natural Language Processing (NLP)

Text-centric tasks: classification, NER, QA, translation, summarization, dialogue. LLMs unify many of these with a single generate-next-token head plus prompting. Classical pipelines used token features + linear models; today embeddings + transformers dominate.

Computer Vision (CV)

Pixels → semantics: classification, detection, segmentation, OCR, depth, pose, video. CNNs were standard; Vision Transformers (ViT) and multimodal LLMs map images to a shared space with language for captioning, VQA, and agents that “see.”

Paradigm You provide Model learns
SupervisedLabeled (x → y) pairsRegressor, classifier, seq2seq
UnsupervisedUnlabeled dataClustering, topic models, VAE, SSL pretext
Reinforcement (RL)MDP, actions, reward signalPolicy to maximize return (used in some LLM “preference” training)

Model lifecycle & evaluation (quick ref)

Split Train / validation / holdout test (never tune on test). For time series, no random shuffle across time without care.
Metrics Classification: precision, recall, F1, ROC-AUC, calibration. Regression: MAE, RMSE, R². Imbalanced data: don’t fixate on accuracy.
Generative eval LLMs: win rates (human/LLM judge with caveats), task-specific gold sets, safety checks, latency + cost per good answer.

2. Generative AI Models

Foundation models: pretrained at scale, then specialized (instruction tuning, tools, fine-tune, adapters). “Open weight” ≠ “open data”—check licenses (Llama, Mistral) vs. closed API-only (frontier hosted).

  • GPT OpenAI Generative Pretrained Transformer. Broad ecosystem: function calling, structured outputs, vision where offered. Compare by version string, not “GPT” alone—capabilities shift per release.
  • Claude Anthropic. Often chosen for long context, nuanced refusals, and coding. Multimodal tiers vary; align pricing with input vs output token mix.
  • Gemini Google: consumer API vs Vertex AI (enterprise controls). Strong multimodal story; “grounding” to Search where supported—understand freshness and what is actually retrieved.
  • Llama Meta open-weight stack for self-host, on-device, distillation, or LoRA fine-tune. You own ops: GPUs, vLLM/TGI, safety filters, and compliance.
Modality family Examples / notes
Text LLMs Chat, code, tool agents, translation. Dominant commercial interface is token-metered API.
Image gen Diffusion models (DALL·E, SD, Imagen, FLUX), editing (inpaint), style control, LoRAs, safety/NSFW policies.
Audio / speech ASR (Whisper-class), TTS, music—watch licensing for voices and music.
Video & 3D Emerging, compute-heavy, higher abuse risk—governance and watermarking are active areas.

Tip: Pin model name + version + region, set temperature/top-p defaults per use case (creative vs deterministic), and log prompts/responses with policy-compliant redaction.

3. Transformer Architecture

Attention is all you need (2017): replace recurrent recurrence with parallelizable attention—why LLMs scale on GPUs. Most chat models are decoder-only (GPT-style); T5/encoder–decoder is still used in some non-chat pipelines.

Concept What it is
Self-attention For each token, score compatibility with every token, softmax weights, value mixture. Scaled dot-product stabilizes gradients.
Multi-head Many parallel heads → diverse relation types; projected and merged. Grouped-query attention (GQA) in some LLMs reduces KV memory.
Tokens Subword pieces (BPE, SentencePiece, WordPiece). Long words become multiple tokens. Use vendor tokenizers; don’t guess “word count = tokens.”
Embeddings Token → vector lookup + positional info. Sentence / passage embeddings for retrieval are often a separate encoder or layer.
Positional Absolute, relative, or RoPE (rotary) so position affects attention scores sensibly in long context.
KV cache At decode time, cache keys/values for past tokens to avoid O(n²) recompute on prefix—critical for inference cost and long chats.
Pretrain → align Causal next-token (pretrain) → SFT (instruction) → RLHF / DPO / RLAIF (preferences) for helpfulness and policy compliance.

Inference knobs (vendor-specific): temperature (diversity), top-p / top-k (truncate tail), max output tokens, and stop sequences. “Reasoning” models may chain internal steps; expose user-visible only what you need.

4. Prompt Engineering Basics

Good prompts: clear role, success criteria, format, edge cases, and (when needed) grounding context. Test with realistic inputs, not toy sentences.

Zero-shot
Instruction only. Works when task is standard and the model can infer format. Ask for a schema (e.g. JSON with keys) to reduce shape errors.
Few-shot (in-context)
Show examples (input → output) to lock down format and nuance. Cost/latency rise with size; choose diverse exemplars over duplicates.
Chain-of-thought (CoT)
Explicit intermediate steps before the answer—helps arithmetic, logic, planning. Beware: longer CoT = more tokens; for production, validate final answer (code exec, tool check) when possible.

More techniques

  • System vs user delimiters — keep durable rules in system (or “developer”) if your platform supports it; put volatile data in user turns.
  • Role prompting — “You are a careful financial analyst; cite line items when uncertain” sets tone and risk posture.
  • Tool / function calling — let the model emit structured function calls for APIs, not paste secrets into prose.
  • ReAct-style — interleave thought → action (tool) → observation in agent loops; cap iterations to avoid runaway cost.
  • Self-consistency / voting — sample N CoT answers, majority vote on multiple-choice—expensive but robust for some benchmarks (not all prod paths).

Quick pattern: (1) Goal → (2) Context → (3) Rules → (4) Output format → (5) Question. Repeat the output contract at the end of long inputs.

5. AI APIs & Integration

You’re not just “HTTP to a model”—operational concerns: org boundaries, quotas, batch vs sync, data handling, and regional deployment.

Provider Highlights
OpenAI Chat Completions / Responses; Assistants (threads, files); embeddings; realtime and batch endpoints where available. Function calling with JSON schema.
Anthropic Messages API; long context tiers; tool use. SDKs designed for tool orchestration. Compare prompt caching features vs plain replay.
Google AI Studio / Vertex Consumer-friendly Studio vs Vertex (Cloud IAM, VPC-SC, enterprise SLAs). Grounding, batch predict, and multimodal inputs vary by model.

Integration checklist

  • Auth & secrets — server-side only; use secret managers, rotate keys, separate dev/stage/prod.
  • Retries — exponential backoff for 429/5xx; respect Retry-After when present; unique idempotency keys for user-visible effects.
  • Streaming — token SSE for chat UX; if you must log, apply redaction/PII rules first.
  • Guardrails — pre-filter inputs (injection, PII) and post-filter outputs; blocklists are brittle alone—layer defense.
  • Cost controls — max output tokens, model tier per route, monthly org caps, kill switches, abuse rate limits on public endpoints.

Prompt injection: treat untrusted user text as data, not as instructions, when the model also sees tool results or system prompts—use structured boundaries and least-privilege tool scopes.

6. RAG & Vector Databases

RAG = retrieve evidence → constrain generation. Good when the knowledge is in docs you control and changes often (policies, SKUs) vs. static pretraining cutoffs.

  1. Ingest: parse PDFs/HTML; normalize tables; tag with tenant and ACLs.
  2. Chunk: 256–1024 token windows common; overlap to avoid cut mid-sentence; keep headings in chunk metadata for provenance.
  3. Index: embeddings (same model for query & index); add keyword/BM25 for hybrid if exact SKUs/IDs matter.
  4. Query: expand query, retrieve top-k, optionally rerank (cross-encoder) before prompt assembly.
  5. Generate: policy prompt + “answer only from context; say if unknown,” with citations to chunk IDs for audit.
Topic Notes
Embeddings If you switch embedding models, reindex everything. Dimensions and distance (cosine vs IP) must match the DB.
Vector DBs Pinecone, Weaviate, Qdrant, Milvus, Chroma, pgvector in Postgres, etc. Need filters? Pick something with rich metadata and composite indexes.
ANN search HNSW, IVF, PQ: tune recall vs QPS. High volume → batch embedding jobs; async refresh when docs change.
Failure modes “Wrong chunk retrieved” → wrong answers confidently. Mitigate: better chunking, rerankers, ask-back questions, user-visible citations, eval suite on your own docs.

When to fine-tune vs RAG: RAG for updatable knowledge; SFT/LoRA for consistent style, format, or domain phrasing; often use both.

7. Responsible AI

Proportionality: match controls to harms and stakes. A draft email assistant ≠ medical triage; scope your governance accordingly.

  • Bias & fairness — Measure slice metrics (region, language, user cohort). Representation bias in data propagates. Avoid “fix in prompt only” for systemic gaps—collect targeted eval sets and involve domain experts.
  • Safety & misuse — Classifier layers, allowlists, abuse detection, and policy enforcement. Red-team new surfaces (uploads, tool calls). Hallucination is an epistemic risk, not a moral one—treat with UX (cites, confidence, human review).
  • Privacy & data rights — Minimize PII, encrypt at rest/in transit, DPA with vendors, regional routing. Be explicit about retention, training use, and user controls (opt-out) per policy.
  • Accountability & transparency — Model name/version in logs, change logs for prompts, incident runbooks, who owns override decisions. RAG: surface sources to users; don’t overclaim “our AI thinks.”
  • Environmental & cost ethics — Bigger model ≠ always better. Right-size, cache, batch, and design flows that don’t autogenerate on every keystroke for sustainability and bills.

Responsible AI is operational: pre-release eval + in-prod monitoring (drift, spikes in refusals, toxic outputs) + regular red-teaming after model upgrades.

Ship checklist (high level)

  1. Define the decision the software makes vs what stays human (especially legal/medical/financial).
  2. Build a test matrix: happy paths, edge cases, adversarial inputs, multilingual if relevant.
  3. Document data flow: what is sent to third-party APIs, where it is stored, and retention length.
  4. Create rollback for model/prompt changes; keep prior versions addressable in logs.
  5. Review access control for RAG corpora: retrieval must respect the same per-user permissions as source systems.

System design & troubleshooting

Patterns for shipping LLM features: when something breaks, narrow whether it is retrieval, model capability, tool wiring, or policy—not “the AI is bad” generically.

Smell Likely cause What to try
Confident but wrong on facts Hallucination, stale prior, no grounding. RAG with citations, lower temperature, ask model to list uncertainty, block critical claims without a tool check.
Refuses too often / too rarely System prompt, safety classifiers, or policy misalignment to use case. Separate internal vs user-facing system instructions; add domain-specific allowlists; log refusal codes.
Omitting key doc sections Chunking splits tables; wrong top-k; embedding mismatch to query type. Parent-child chunks, title-aware splits, query expansion, hybrid search, reranker after ANN.
Intermittent tool errors Unvalidated JSON, schema drift, unhandled API errors, timeouts. Validate tool args with a schema, mirror OpenAPI, circuit-break, surface errors back to the model in <observation> fields.
Cost spikes / slow UX Oversized contexts, N-shot bloat, repeated system prompt, wrong model tier. Prompt caching, trim history, cache embeddings, use smaller model for triage, batch offline jobs.

Observability: log trace IDs from UI → retriever → LLM → tools; store prompt hash + model + params for reproducible debugging. Never log raw PII in shared analytics without policy.

Reference — acronyms & one-liners

LLMLarge language model—usually decoder-only transformer trained to predict the next token.
SFTSupervised fine-tuning on instruction/answer pairs to teach format and “assistant behavior.”
RLHFReinforcement learning from human feedback—trains a reward or policy from preference rankings.
DPO / RLAIFPreference learning alternatives (direct preference optimization, AI-labeled feedback) to align models.
RAGRetrieval augmented generation: fetch docs → add to context → answer with grounding.
LoRALow-rank adaptation—small trainable adapter matrices; efficient fine-tuning of large models.
GQAGrouped-query attention—share KV across heads to cut memory in long sequences.
ANN / HNSWApproximate nearest neighbor search; HNSW is a common graph index for vectors.
SSEServer-Sent Events—one-way server→browser stream; many LLM HTTP APIs use SSE for token-by-token output.
PII / GDPRPersonally identifiable information; European privacy regulation—design controls, not just disclaimers.
OCR / ASR / TTSOptical character recognition; automatic speech recognition; text-to-speech—typical I/O for multimodal stacks.
SLA / SLOService-level agreement vs objective—set latency/error budgets for AI features with human fallback when breached.
A/B or banditRun prompt/model changes behind experiments; measure business metrics, not only perplexity or bench scores.
MLOps / LLMOpsOperational discipline for model lifecycle: data versioning, eval registries, prompt/version promotion, and rollback.
NLP / NLU / NLGNatural language processing / understanding (intent, entities) / generation (surface text from meaning).
MoEMixture of experts—routes tokens to subnetworks; increases capacity without every parameter firing each forward pass.
CWE / STRIDEVulnerability and threat models—map AI features (RAG, tools, file uploads) to known abuse and injection classes.
Tokens / TPMBilling units; tokens-per-minute (or concurrent request) rate limits on APIs—architect for backoff and sharding.

Myths: “bigger context removes need for RAG” (retrieval still wins on precision); “zero temperature is always right” (some tasks need diversity); “the model read my whole PDF” (only what you pass in the window).
