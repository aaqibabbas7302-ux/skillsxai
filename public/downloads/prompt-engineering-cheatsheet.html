Prompt Engineering Cheat Sheet

A concise reference for designing reliable prompts: structure, examples, reasoning, system behavior, sampling, and practical templates—optimized for screen and print.

At a glance

  1. Define role + goal + constraints before the task.
  2. Choose zero-shot for breadth; few-shot for rigid patterns.
  3. Use CoT when steps matter; use JSON / keys when parsing matters.
  4. Put system rules once; keep user turns for data + ask.
  5. Low temperature for facts/code; raise for brainstorming—after the prompt is clear.

1. Prompting fundamentals

Strong prompts usually combine four ingredients. Order them so the model sees who it is, what it knows, what to do, and how to answer.

Element Purpose Prompt fragment
Role Sets expertise and tone. You are a senior data analyst…
Context Grounds facts, audience, constraints. Our product is B2B SaaS; readers are PMs…
Task Single, testable instruction. Compare options A vs B for migration risk.
Format Defines structure, length, and fields. Reply with: Summary / Risks / Next steps (bullets).
  • One job per prompt: split “research + draft + critique” into separate turns or sections.
  • Be specific about success: “actionable for a junior dev” beats “make it good.”
  • Ask for uncertainty: “List assumptions and what would change your answer.”

Delimiters & structure

Use clear boundaries so the model does not mix instructions with data. XML-style tags, markdown headings, and triple-quoted blocks all work; stay consistent within a project.

Pattern When to use
<document> … </document> Long source text: summarize, extract, or quote from inside the tag only.
<rules> … </rules> Non-negotiables the assistant must re-read (legal, style, PII handling).
--- USER_INPUT --- Simple, visible separation when XML is awkward; good for comments from users.
# Minimal structured prompt (inline)
<role> You are a careful reviewer. </role>
<task> List gaps in the argument. </task>
<document> … paste text … </document>

2. Zero-shot vs few-shot

Zero-shot: no examples—just instructions. Few-shot: 1–10 (often 2–5) input–output pairs that show the pattern you want.

Dimension Zero-shot Few-shot
Best for General tasks, speed, large context for documents Strict format, classification, extraction, style mimicry
Trade-off May guess format unless you specify it Uses tokens; bad examples teach bad habits
Tip Add format schema + criteria in text Keep examples diverse, label edge cases, avoid contradictions
Zero-shot example

Classify the following ticket as billing, bug, or feature. Output JSON: {"label": "..."}. Ticket: …

Few-shot pattern

Input: … → Output: {"label":"bug"}
Input: … → Output: {"label":"billing"}
Input: … → Output:

Negative (contrast) examples

Showing what not to do (wrong label, bad tone, disallowed PII) can reduce repeated mistakes—keep negatives short to save tokens.

  • Pitfall: contradictions between positive and negative demos confuse the model—align on one rubric.
  • Rule of thumb: if the task is “same as before but in French,” a single positive pair often beats a long explanation.

3. Chain-of-thought (CoT) prompting

CoT asks the model to show step-by-step reasoning before the final answer. It often improves math, logic, planning, and multi-step analysis—especially in larger models.

  • Zero-shot CoT: append “Let’s think step by step” (or “Explain your reasoning, then conclude”).
  • Few-shot CoT: each demo includes visible reasoning + final answer—match the style you want at test time.
  • When to skip: latency/cost-sensitive paths, or when you need only a label and intermediate text could derail parsing—use structured output instead.
# Template
Question: …
Before answering, work through: givens → missing info → options → recommendation.
End with a Final answer: line.

Worked micro-example (no calculator)

Q: A bat and ball cost $1.10 total. The bat costs $1 more than the ball. What does the ball cost?
# Ask for explicit steps, then a single numeric final line for parsing.
Reasoning: Let ball = b, bat = b+1. b+(b+1)=1.10 → 2b+1=1.10 → b=0.05.
Final answer: $0.05

4. System prompts

The system message (where your API supports it) sets durable defaults: persona, safety and policy, tooling or data rules, and output shape. User messages carry the task-specific request.

Layer Include
Persona Role, seniority, tone (neutral, concise, teaching).
Constraints Refuse unsafe asks; cite when unsure; no PII; locale/units.
Output format Markdown vs JSON; required keys; “no preamble” for machine parsing.
Tool & data rules Which tools are allowed, required JSON shape for function calls, when to say “I don’t know” vs fetch.

If you only have a single user blob, mimic a system block with a clear header: [System] … [User] …

# Example system (abbreviated)
You are a support lead for Acme. Be concise, polite, and proactive.
Never invent order IDs or policy exceptions. If a customer shares card numbers, refuse and suggest secure channels.
For every answer: (1) Diagnosis (2) Next step the customer can try (3) Escalation path if unresolved.
Default tone: empathetic, no blame.

5. Temperature & sampling parameters

These knobs change how tokens are picked, not what the model “knows.” Names vary by provider; behavior is broadly similar.

Parameter Effect Typical use
temperature Higher → more randomness; lower → more deterministic. ~0–0.3 facts, code, extraction; 0.7–1 creative drafts (if not using top_p alone).
top_p Nucleus sampling: limit to smallest set of tokens whose cumulative prob ≥ p. Often 0.9–0.95; combine carefully with temperature—many apps fix one and tune the other.
max_tokens / length Hard cap on generation length (output side). Set high enough for CoT + answer; too low truncates reasoning. Watch total context window.
stop sequences End generation when a token sequence appears—useful for strict templates. e.g. stop at </json> or a sentinel line ### END.
frequency / presence * *(Provider-dependent) Nudge to repeat phrases less or to encourage new topics. Reduce repetition in long lists; small tweaks—avoid large values that break coherence.
seed When supported, fixes stochastic sampling for comparable runs. QA, regression on prompts, and A/B tests of wording at fixed temperature.
  • Reproducibility: use low temperature and fixed seed if the API exposes seed.
  • Don’t chase randomness for accuracy—clarify the prompt and ground with retrieval first.

6. Advanced techniques

Self-consistency

Sample multiple CoT chains (higher temperature), then majority-vote the final answers or aggregate rationales. Better on hard reasoning; costs more tokens.

Tree-of-thought (ToT)

Explore branching plans—decompose, score intermediate steps, prune bad branches. Useful for puzzles, design, or search-like planning; implement as multiple LLM calls or explicit graph prompts.

ReAct (reasoning + action)

Interleave Thought → Action → Observation loops: the model plans, calls tools (search, code, API), reads results, updates. Foundation for many agents; requires strict tool schemas and stop conditions.

Thought: I need 2024 revenue, not a guess.
Action: company_lookup("ACME", field="revenue_2024")
Observation: { "revenue_2024_usd": 12500000 }
Thought: I can now compare YoY if user gives 2023…

Program-aided (PAL / tool-as-code)

Delegate arithmetic or data transforms to a real runtime: the model writes code, you execute, return stdout as Observation. Reduces “mental math” errors in LLMs on large or precise calculations.

Reflexion & critique

After a first answer, run a critic pass: “List errors, missing edge cases, then rewrite.” Cheap way to add robustness when multi-chain voting is too costly.

7. Reusable prompt templates

Replace bracketed fields. Keep system-level rules separate from the task line.

Writing

Audience: [role] · Goal: [outcome] · Voice: [adjectives] · Length: [words] · Structure: [outline] · Avoid: [list]. Draft [artifact], then give 2 revision options for tone.

Coding

Stack: [lang/framework] · Task: [one sentence] · Constraints: perf, style, deps · Deliver: (1) plan (2) code blocks with paths (3) tests or manual QA steps (4) known risks.

Analysis

Data: [paste or describe] · Question: [hypothesis] · Method: [metrics/comparisons] · Output: key findings, chart suggestions, limitations, follow-up data needs.

Summarization

Summarize for [audience] in [N] bullet points. Preserve numbers and proper nouns. Flag contradictions. End with: Decisions, Open questions, Action items (owner, due).

Evaluation (rubric)

You are scoring [artifact] for [purpose]. Rubric: (1) Correctness 0–2 (2) Clarity 0–2 (3) Safety 0–2. For each, quote evidence, then give total /6 and the single highest-impact fix.

Debugging a bad output

Here is the prompt: … Here is the bad output: … List (a) which instruction was violated (b) missing context (c) a revised prompt under 150 words. Do not fix the content—only the instructions.

JSON extraction

From the text in <content>, extract: {"people":[],"dates":[],"amounts":[],"open_questions":[]}. Return valid JSON only. If a field is empty, [].

8. Common mistakes & fixes

Mistake Fix
Vague goal (“improve this”) Name metric: clarity, brevity, conversion—give before/after example if possible.
Stacking many tasks in one message One primary objective per turn; chain prompts or use a checklist the model must complete in order.
Context window overflow Summarize long docs first; retrieve top chunks; put irreducible facts up front; reduce few-shot count.
Ignoring parse format Request JSON with keys + one full example; say “return ONLY valid JSON, no markdown fence.”
Blind trust Ask for citations, self-checks, or independent verification for high-stakes outputs.
Burying the lede (task last) Repeat the key ask at the end, or use headings so the last line is the deliverable: “Output only the table.”
Inconsistent few-shot quality Manually review each example; drop noisy rows; add one “hard” but correct example.
Overfitting temperature to fix prompts If answers swing wildly, first sharpen instructions and add constraints—then re-tune sampling.
Lost-in-the-middle (long context) Place critical facts at the start or end; or chunk + map-reduce with explicit merge rules.

Iterative improvement loop

  1. Write a minimal prompt that hits role + task + format.
  2. Run 3–5 real inputs; collect failure modes (not vibe complaints).
  3. Patch one issue per revision (e.g. “always quote line numbers” OR “use ISO dates”).
  4. Add few-shot or CoT only if instructions alone are insufficient.
  5. Version prompts in source control: prompts/support/v3.md beats untracked copy-paste.
